## 1. What this project is

A public tracker for comment letters submitted on the SEC's proposed rule **S7-2026-15** (Optional Semiannual Reporting by Public Companies — replaces quarterly Form 10-Q with optional semiannual Form 10-S).

The public comment period is open ~60 days from May 2026. The tracker collects, classifies, and visualizes every letter as it arrives.

**Entity (three-rater LLM ensemble)** — done

**Rationale (20-code taxonomy, three-rater LLM ensemble)** — done
- Multi-label per letter (each letter can invoke 0+ codes); 159/159 letters carry at least one code
- 16 SEC-engaged codes, 3 commenter-distinctive codes (IP, US, RI), 1 "no rationale" code (NR)
- Taxonomy anchored on the SEC's proposing release (33-11414) with verbatim quotes for each SEC-engaged code
- **Three-rater LLM ensemble (Claude Opus 4.7, three rubric prompts):**
  - Rater 1 Primary (balanced) / Rater 2 Literalist (strict, surface-only) / Rater 3 Inclusive (expansive)
  - Aggregation: per-(letter, code) majority vote → public-facing `rationales`
  - Agreement: 86 unanimous (54.1%) / 61 majority (38.4%) / 12 split (7.5%)
  - **Mean per-code Cohen's κ across pairs: 0.813** (substantial; in line with multi-prompt LLM-annotation benchmarks)
  - Per-code κ ranges from 0.22 (OV) and 0.33 (SG, both near-zero frequency) up through 1.00 (ICc, CC, ICs, all unanimous). Surface-readable codes high: US 0.95, FR 0.93, AL 0.91; inferential codes lower: RI 0.76, CMP 0.68.

- Two SEC-engaged codes fire near-zero in commenter letters: SG (signaling, 0-1), OV (option value, 0-2). The SEC engages these in its own economic analysis but commenters don't engage them as standalone arguments.
- **Cross-model validation (ChatGPT-5.5, n=137 overlap):** GPT-Majority vs Claude-Majority mean per-code Cohen's κ = 0.477 (moderate). Set-level exact match 24.8%; mean Jaccard 0.52. ### Regression — done
- Logit: Support=1 / Oppose=0 (Conditional dropped), 7 entity dummies (reference: Individual) + log(words+1)
- **N = 141** letters in sample (6 Support, 135 Oppose), **McFadden pseudo-R² = 0.250**
- Four buckets have 0 Support letters and are flagged "separated" (Accountant CPA, Issuer-former, Investment professional, Industry practitioner)
- **Student bucket: significant at p = 0.027** (β = +3.44); separated buckets stay separated
- Script: `_scripts/run_regression.py` (handles singular-Hessian via BFGS fallback and detects separation from the data directly)
- **Cross-classifier sensitivity:** rerunning the regression with each of 8 stance classifiers (Claude P/L/S/Majority + GPT P/L/S/Majority) confirms the substantive direction is invariant — Student, Issuer-current, Academic all carry positive coefficients across every classifier; the four separated buckets stay separated. Which single bucket crosses p<0.10 shifts (Student under Claude-Majority; Issuer-current under GPT-Majority). Sensitivity footnote pending next regression-panel touch.

## 5. Pipeline (Option A2 — GitHub Actions, shipped 2026-05-16)

SEC docket sync runs as a GitHub Action on GitHub's servers, twice daily at 14:00 UTC (10 AM ET) and 23:00 UTC (7 PM ET). The Action:

1. Checks out the repo.
2. Runs `_scripts/daily_fetch.py`, which hits SEC.gov pages 1-3 of the docket index, diffs URLs against `_meta/renumbered_records.json`, fetches the body for each new URL, and appends a placeholder record with stance / entity / rationales all set to "Unclassified".
3. Regenerates the leakage-free `Letters_<N>_for_models/` mirror.
4. Re-runs the regression comparison (`run_regression_compare.py`).
5. Rebuilds `public_site/` via `build_and_push.py --no-push`. The build's `build_snapshot` filter excludes Unclassified rows so the public table only shows fully-classified letters.
6. Commits and pushes back to `main`. GitHub Pages auto-deploys.

Workflow file: `.github/workflows/daily-fetch.yml`. Runs as user `sec-tracker-bot`. Triggerable manually from the Actions tab via "Run workflow".

Tzachi opens Cowork when convenient, sees "X letter(s) awaiting Claude classification" in the local sync output, and Claude runs the three ensembles on those placeholders. This is the editorial-control flavor (A2) — the Action only does fetch + placeholder + build, never auto-classification.

Mac side: `_scripts/sync_from_github.sh` pulls the canonical files (`renumbered_records.json`, `regression_compare.json`, `public_site/*`) and any new `Letters/NN_*.md` files back to the Dropbox folder. Can be wired into launchd or run on demand before a Cowork session.

**Docket index URL:** `https://www.sec.gov/rules-regulations/public-comments/s7-2026-15`. Drupal-based, paginates via `?page=N` (zero-indexed). Letter URLs in the table point at the older `/comments/S7-2026-15/...` path. The fetcher adds a unix-timestamp `_t={epoch}` query parameter to bypass CloudFront's URL-level cache (the same CDN-stale-page-0 issue that broke the prior Cowork fetch task — STATUS §3). Crawls 8 pages to cover roughly a week of headroom against the ~30 letters/day peak rate.

**Auth.** Supabase Auth with magic-link email sign-in. Already on the project; free tier covers 50,000 MAU.

**Registration form** collects, with email already captured by Auth:

**Schema sketch** (additive — does not break the existing `votes` table in `_meta/supabase_setup.sql`):

```sql
-- New profiles table, 1:1 with auth.users
create table profiles (
  user_id            uuid primary key references auth.users(id) on delete cascade,
  name               text not null,
  institution        text not null,
  role               text not null check (role in (
                       'academic_researcher','accountant_cpa',
                       'issuer_current','issuer_former',
                       'investment_professional','industry_practitioner',
                       'student','individual','other')),
  registration_path  text not null check (registration_path in ('self_service','invite')),
  invite_code_used   text,  -- nullable; only for invite path
  created_at         timestamptz not null default now(),
  email_verified_at  timestamptz
);

alter table profiles enable row level security;
create policy "profile_self_read"   on profiles for select using (auth.uid() = user_id);
create policy "profile_self_insert" on profiles for insert with check (auth.uid() = user_id);
create policy "profile_self_update" on profiles for update using (auth.uid() = user_id);

-- Optional invite_codes table (for the invite path; self-service does not touch it)
create table invite_codes (
  code         text primary key,
  email_hint   text,                          -- optional; the email we mailed this code to
  used_by      uuid references auth.users(id),
  created_at   timestamptz not null default now(),
  used_at      timestamptz,
  expires_at   timestamptz
);

-- Extend votes table
alter table votes add column user_id uuid references auth.users(id);
alter table votes add column panel   text not null default 'research';
-- backfill+enforce: existing anonymous rows can stay (panel='legacy'), new rows must carry user_id
create policy "vote_insert_self" on votes for insert
  with check (auth.uid() = user_id and panel = 'research');
```

## 7. Open design decisions / questions parked for later

1. **Whether to push voting to production immediately when Supabase is live, or stage on dev first** — current plan is stage on dev for a week, evaluate, then push.
2. **Anti-spam hardening for crowd votes** — current setup is one-vote-per-browser + 60/hour rate limit. May need captcha or attention checks if abuse arrives.
3. **Per-issue voting questions wording** — six questions currently (cadence-OK, assurance, retail-focus, alt-cadence, cost-burden, SEC-mandate). May refine after seeing early responses.
4. **Whether to surface "Unclassified" letters publicly** under Option A2, or hold them in staging until Claude has classified them.
5. **Whether to promote the IP two-point callout** from dev to production (see §6.5).
6. **Crowdsourced classification — engagement vs scientific rigor.** Opening the site to public classification has two purposes that pull against each other. As engagement, letting visitors tag stance / entity / rationale codes on letters drives time-on-site and gives the project social presence. As science, public-collected annotations face the nonsense-injection problem — bad-faith votes, casual misreads, single-author flooding, and the absence of any rater training. Three design questions to settle before opening up: (a) do we surface public votes alongside Claude / GPT calls publicly, or hold them in a research backstage and only aggregate ones that pass quality gates? (b) what quality gates — minimum letter-read time, attention-check letters seeded in the rotation, two-vote confirmation, IP / browser dedup, captcha for non-trivial volume? (c) how do we frame the call-to-action — "classify a letter, see how your call compares" (entertainment-leaning, low friction, public reveal) vs "join a research panel, get a code-of-conduct briefing, take the rubric" (scientific-leaning, higher friction, recruited audience)? Working hypothesis: keep the engagement path open behind a clear "your-vote-vs-the-algorithm" UI and aggregate only quality-gated votes into any scientific claim. The Carlson & Burbano framework warns about LLM-iteration p-hacking; the crowdsourced version has its own version of this risk (cherry-picking which votes count). Settle the design before turning Supabase on.
